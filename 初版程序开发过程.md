# 初版程序开发过程

## 1. 阅读实验要求

本次实验要求实现一个基于递归下降分析法的表达式计算程序。程序需要能够处理基础数学运算，包括：

1. 加法 `+`
2. 减法 `-`
3. 乘法 `*`
4. 除法 `/`
5. 多层括号 `()`

实验文档中给出了三种候选文法，需要对它们进行比较，并选择一种适合递归下降实现的文法。

## 2. 文法分析与选择

实验中给出的第一种文法为：

```text
E -> T + T | T - T
T -> F * F | F / F
F -> NUM | (E)
```

这种文法结构比较固定，只能描述较短的二元表达式，不适合处理连续运算。例如 `1 + 2 + 3`、`2 * 3 * 4` 这类表达式不能自然地由该文法描述，因此不适合作为本次实验的最终实现文法。

第三种文法为：

```text
E -> T + E | T - E
T -> F * T | F / T | epsilon
F -> NUM | (E)
```

这种文法虽然引入了递归结构，但存在两个问题。首先，`T -> epsilon` 可能导致某些不完整表达式被错误接受。其次，`E -> T + E` 和 `E -> T - E` 是右递归形式，容易使减法等左结合运算被解析成右结合。例如 `10 - 3 - 2` 按常规应理解为 `(10 - 3) - 2`。

最终选择第二种文法：

```text
E  -> T E'
E' -> + T E' | - T E' | epsilon
T  -> F T'
T' -> * F T' | / F T' | epsilon
F  -> NUM | (E)
```

选择原因如下：

1. 没有左递归，适合递归下降分析。
2. `E` 层处理加减，`T` 层处理乘除，`F` 层处理数字和括号，能够自然体现运算优先级。
3. 可以支持连续运算和多层括号。
4. 每个非终结符可以直接对应一个 C++ 函数，程序结构清晰。

## 3. 初始代码框架设计

初版程序使用 C++ 编写，整体分为两个主要模块：

1. `Lexer`：词法分析器，负责把输入字符串拆分成 Token 序列。
2. `Parser`：语法分析器，负责根据文法递归分析 Token，并计算表达式结果。

Token 类型设计如下：

```cpp
enum class TokenType {
    Number,
    Plus,
    Minus,
    Multiply,
    Divide,
    LeftParen,
    RightParen,
    End
};
```

每个 Token 使用结构体保存：

```cpp
struct Token {
    TokenType type;
    double value;
    std::string text;
};
```

其中，`type` 表示 Token 类型，`value` 用于保存数字 Token 的数值，`text` 用于保存 Token 对应的原始文本。

Parser 中的函数与文法对应关系如下：

```text
parseExpression()          -> E
parseExpressionPrime(...)  -> E'
parseTerm()                -> T
parseTermPrime(...)        -> T'
parseFactor()              -> F
```

这种设计的好处是程序结构和文法规则一一对应，便于理解和调试。

## 4. Lexer 实现过程

词法分析器的任务是从输入字符串中识别数字和运算符，并生成 Token 序列。

初步实现时，Lexer 中设计了以下辅助函数：

```cpp
bool isAtEnd() const;
char peek() const;
char advance();
bool isDigit() const;
void skipWhitespace();
Token scanNumber();
```

各函数作用如下：

1. `isAtEnd()`：判断当前位置是否到达输入字符串末尾。
2. `peek()`：查看当前位置字符，但不移动位置。
3. `advance()`：返回当前字符，并将位置向后移动一位。
4. `isDigit()`：判断当前位置字符是否是数字。
5. `skipWhitespace()`：跳过空格。
6. `scanNumber()`：扫描一个数字 Token。

在 Lexer 的调试过程中，发现并修正了几个问题：

1. 运算符分支最初没有把生成的 Token 加入 `tokens`，也没有移动 `pos_`，会导致死循环。
2. 除号最初写成了特殊符号 `÷`，后来改为实验要求的 `/`。
3. 数字扫描时最初直接使用字符的 ASCII 值，导致 `'3'` 被当成 `51`，后来改为 `advance() - '0'`。
4. `switch` 中补充了 `default`，当遇到非法字符时抛出异常。
5. End Token 的 `text` 字段改为空字符串 `""`。

当前 Lexer 已经可以处理整数、空格、四则运算符和括号。

## 5. Parser 辅助函数理解

Parser 中有三个容易混淆的辅助函数：

```cpp
bool check(TokenType type) const;
bool match(TokenType type);
const Token& consume(TokenType type, const std::string& message);
```

它们都和“当前 Token 类型判断”有关，但职责不同：

```text
check   = 只检查当前 Token 类型，不移动位置
match   = 如果匹配，就消费当前 Token；不匹配则返回 false
consume = 必须匹配并消费当前 Token；不匹配则抛出异常
```

其中，`match()` 适合用于可选语法，例如：

```text
E' -> + T E' | - T E' | epsilon
```

因为 `+` 和 `-` 不一定出现。

`consume()` 适合用于必须出现的语法，例如：

```text
F -> (E)
```

当已经读到左括号 `(` 后，后面必须出现右括号 `)`，否则就是语法错误。

## 6. Parser 实现过程

Parser 的入口函数是 `parse()`。它从开始符号 `E` 开始解析：

```cpp
double result = parseExpression();
consume(TokenType::End, "expected end");
return result;
```

其中 `consume(TokenType::End, ...)` 用于确认表达式已经完整结束，防止输入中存在多余 Token。

`parseExpression()` 对应文法：

```text
E -> T E'
```

因此先解析 `T`，再把结果交给 `E'`：

```cpp
double left = parseTerm();
return parseExpressionPrime(left);
```

`parseTerm()` 对应文法：

```text
T -> F T'
```

因此先解析 `F`，再把结果交给 `T'`：

```cpp
double left = parseFactor();
return parseTermPrime(left);
```

`parseFactor()` 对应文法：

```text
F -> NUM | (E)
```

如果当前 Token 是数字，则返回数字值。如果当前 Token 是左括号，则递归调用 `parseExpression()` 解析括号内部表达式，并要求后面必须有右括号。

## 7. 关键错误与修正

Parser 调试时最重要的问题出现在 `parseExpressionPrime()` 和 `parseTermPrime()` 中。

最初错误写法的思路是先解析右侧内容，再判断当前是否有运算符。例如：

```cpp
double right = parseTerm();
if (match(TokenType::Plus)) return parseExpressionPrime(inheritedValue + right);
if (match(TokenType::Minus)) return parseExpressionPrime(inheritedValue - right);
```

这种写法的问题是：当输入为 `1 + 2` 时，解析完 `1` 后，当前 Token 是 `+`。此时如果先调用 `parseTerm()`，`parseTerm()` 会尝试从 `+` 开始解析一个项，从而报错。

正确理解应该是：

```text
E' -> + T E' | - T E' | epsilon
```

也就是说，`E'` 应该先判断当前 Token 是否是 `+` 或 `-`。如果是，再解析右侧的 `T`。

修正后的逻辑为：

```cpp
if (match(TokenType::Plus)) {
    double right = parseTerm();
    return parseExpressionPrime(inheritedValue + right);
}

if (match(TokenType::Minus)) {
    double right = parseTerm();
    return parseExpressionPrime(inheritedValue - right);
}

return inheritedValue;
```

`parseTermPrime()` 也存在同样问题。根据文法：

```text
T' -> * F T' | / F T' | epsilon
```

应该先判断当前 Token 是否为 `*` 或 `/`，如果是，再解析右侧的 `F`。

修正后的逻辑为：

```cpp
if (match(TokenType::Multiply)) {
    double right = parseFactor();
    return parseTermPrime(inheritedValue * right);
}

if (match(TokenType::Divide)) {
    double right = parseFactor();
    return parseTermPrime(inheritedValue / right);
}

return inheritedValue;
```

这个问题说明，在递归下降分析中，函数实现必须严格按照文法右部符号的顺序进行。

## 8. 当前测试结果

修正后，对程序进行了初步测试：

```text
输入：3 * (4 + 2) - 10 / 2
输出：13

输入：1+2*3
输出：7

输入：10-3-2
输出：5

输入：8/2/2
输出：2
```

这些结果说明当前程序已经能够正确处理整数四则运算、括号、运算优先级和左结合关系。

## 9. 当前版本的不足

当前版本仍然有一些可以继续完善的地方：

1. `scanNumber()` 目前只支持整数，还不支持小数。
2. `skipWhitespace()` 目前只跳过普通空格，还不支持制表符等其他空白字符。
3. 数字 Token 的 `text` 字段当前没有保存真实原始文本。
4. 错误信息还比较简单，可以进一步改得更明确。
5. 还没有支持变量赋值，这是实验中的进阶要求。

## 10. 阶段总结

初版程序已经完成了表达式计算器的基本结构，包括词法分析器、语法分析器以及基于第二种文法的递归下降分析流程。

在实现过程中，最重要的收获是理解了文法和代码函数之间的对应关系：

```text
E  -> parseExpression()
E' -> parseExpressionPrime()
T  -> parseTerm()
T' -> parseTermPrime()
F  -> parseFactor()
```

同时也理解了递归下降分析中 `check()`、`match()` 和 `consume()` 的区别，以及 `E'`、`T'` 这种“后缀递归”函数应该如何接收并继续处理已经计算出的左侧结果。
